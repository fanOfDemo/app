使用System.nanoTime测试代码执行时间

发表于 Nov 2 2016 |

之前写插件下载的过程中遇到过一个很细节的知识点。

业务场景是这样的，APP启动后的第一时间做检查并下载新版插件，比如当前网速是每秒20k而插件的大小是600K,意味着需要等待30s用户才能做下一步操作，很明显这个时间是非常不合理的。有什么办法做一个超时触发？当然有！

于是，我在开始下载的位置获取了系统的时间戳，这个时间戳是从1970年开始算的毫秒时间戳，并且众所周知这个时间戳可以转换成对应的日期小时分钟秒毫秒

long startTime = System.currentTimeMillis;

在每次读取服务器文件流时，做了一个减法计算耗时，

long delyTime = System.currentTimeMillis - startTime

判断delyTime是否大于20s，如果大于20s，触发一个下载超时，取消本次更新。

OK，从上文看，没有任何问题，非常完美解决了问题。

大神说了，看优秀框架的源码才能真正进步。前些天不小心看了Picasso的源码，看到into(Callback callback)方法里，RequestCreator的fetch()方法有这么一行，奇怪这玩意是啥，

long started = System.nanoTime();

然后Request有这么一个方法， 差不多懂了

private static final long TOO_LONG_LOG = TimeUnit.SECONDS.toNanos(5);

String logId() {
    long delta = System.nanoTime() - started;
    if (delta > TOO_LONG_LOG) {
          return plainId() + '+' + TimeUnit.NANOSECONDS.toSeconds(delta) + 's';
    }
    return plainId() + '+' + TimeUnit.NANOSECONDS.toMillis(delta) + "ms";

  }

进一步阅读java API发现：

public static long nanoTime()

返回最准确的可用系统计时器的当前值，以毫微秒为单位。

此方法只能用于测量已过的时间，与系统或钟表时间的其他任何时间概念无关。返回值表示从某一固定但任意的时间算起的毫微秒数（或许从以后算起，所以该值可能为负）。此方法提供毫微秒的精度，但不是必要的毫微秒的准确度。它对于值的更改频率没有作出保证。在取值范围大于约 292 年（263 毫微秒）的连续调用的不同点在于：由于数字溢出，将无法准确计算已过的时间。

例如，测试某些代码执行的时间长度：

long startTime = System.nanoTime();
// ... the code being measured ...
long estimatedTime = System.nanoTime() - startTime;

返回：
系统计时器的当前值，以毫微秒为单位。注意这里说的豪微秒，也就是纳秒，

到这里发现前面觉得完美的代码有什么问题了吗，没错将日期对应的毫秒时间戳改成精确计时的纳秒时间能更准确计算出代码执行的耗时。

距离上次写文章有好一段时间了，之前讲过设计模式和插件化，今天来分享个实用的东西。

首先先设计出SDK和宿主的公共部分，目前ZeusPlugin插件化只能得到插件的MainClass,ZeusPlugin的这个class是个activity，可我的SDK不是activity怎么调用这个类以及这个类的方法内，有两个办法，

  • 1：通过反射调用这个类的方法，然后使用反射传参

  • 2：使用公共接口，宿主和插件都能识别的接口（interface）

  • 反射：缺点和解决方法：假如我们的SDK的接口方法比较多，使用反射明显是非常低效的，代码量大（工具类可以减少代码量），性能不高（做点cache），插件容易不小心改动接口代码。

  • 接口：只需要一行代码就可以获取到接口的实例，无反射直接调用实例方法，接口固定难以更改。

以上是准备框架和理论工作，现在回到业务，假如我们现在有两个游戏平台的业务，这里叫A游戏，B游戏
，他们都有固定的接口对外，但是内部实现不一样，这个时候，我们先抽象一个接口

public interface IGameAPI {

    void init (JSONObject jsonObject, ValueCallback<JSONObject> valueCallback);

    void login (JSONObject jsonObject, ValueCallback<JSONObject> valueCallback);

    void pay (JSONObject jsonObject, ValueCallback<JSONObject> valueCallback);

    void logout (JSONObject jsonObject, ValueCallback<JSONObject> valueCallback);

}

公共接口一开始可以写在sdk的实现module中，代码写完后将该接口文件移到zeusPlugin中，将zeusPlugin打包成jar，提供给SDK module以provided形式依赖（只参与编译不参与打包），同时作为宿主的compile(可作为编译及打包),这样插件和宿主都能识别使用共同的接口，之后可以将sdkmodule中的相关公共接口代码注释或删除，并将接口的实现改成jar中的接口，宿主负责调用，插件负责实现，从而实现了SDK的热更新，注意这个接口要保证尽可能兼容到所有的业务功能，不然这个热更新就没什么意义了。

然后再写实现公共接口的同时，增加静态类单例模式（为什么使用单例、有什么坑稍后再讲），

andorid插件化研究

发表于 Jul 7 2016 |

一、奇虎开源插件化框架（基于Hook PMS AMS的launcher思路）

针对奇虎开源插件化框架的基础讲解Hook 静态代理，动态代理，PMS，AMS
源码及hook思路讲解 

优点：
宿主插件完全分离，跨进程的沙箱环境，基于launcher的启动流程，通过欺骗系统的启动流程的思路启动插件apk，于系统来说宿主就是一个launcher，相对于代理的方式更可靠安全
缺点：
1、不支持插件的notifycation
2、需要框架的使用者实现宿主和插件的路由协议，以便跨进程通讯

二、百度的静态代理实现的动态加载（静态代理）

优点：易懂、上手简单、适合轻量级插件
缺点：
1、插件代码需要支持that语法以供宿主识别
2、不支持插件中的数据库
3、插件中that语法在复杂的activity继承关系中难以维护
4、不支持activity和fragmentActivity以外的activity，手动增加其它activity的支持会造成大量的代理，额外增加大量内存消耗以及不可控性，不适合较大的插件

三、利用代码生成机制动态生成代码的插件化方案（动态代理及动态代码生成）

原理和上面的静态代理类似，不同之处在于上面使用手动代理并加载插件中activity，此框架利用dexmaker框架实现了复制插件代码并在宿主中生成二进制的代码以实现插件化

四、android-dynamical-loading
疑似来自B站的组件化下载及行为规范方案，插件化未完成（目前是实现了插件的下载和校验，以及尝试对插件和宿主制定统一约束和规范）

只做了基础的组件化方案，框架本身并不支持动态加载activity和service等等，如框架作者所说，此框架适合设计自定义的组件化包，比如脱离android生命的类，无acitivty或service的sdk等等，比如B站的播放器核心so包的频繁更新适合用于动态加载，话说回来，适不适合技术说了也没用啊，产品就要这么干也没办法

本节主要讲解各种单例模式的使用以及优缺点

1懒汉单例：

特点：
1、线程不安全
2、类加载时不创建实例，运行时获取单例，类加载快，运行时慢

public class Singleton1 {
    private static Singleton1 instance;
    private Singleton1(){}

    public static Singleton1 getInstance() {
           if (instance == null) {
            instance = new Singleton1();
        }
        return instance;
    }
}

2饿汉单例

特点：
1、线程不安全
2、类加载时创建实例，加载时慢，运营时快

public class Singleton2 {
    private Singleton2 () {
    }

    private static Singleton2 instance = new Singleton2 ();

    public static Singleton2 getInstance () {
        return instance;
    }
}

3双重校验锁

特点：1、线程安全

public class Singleton4 {
    private volatile static Singleton4 singleton;
    private Singleton4(){}
    public static Singleton4 getSingleton() {
        if (singleton == null) {
            synchronized (Singleton4.class) {
                if (singleton == null) {
                    singleton = new Singleton4();
                }
        }
    }
    return singleton;
    }
}
