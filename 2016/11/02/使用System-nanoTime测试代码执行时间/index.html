使用System.nanoTime测试代码执行时间

发表于 Nov 2 2016   |  

之前写插件下载的过程中遇到过一个很细节的知识点。

业务场景是这样的，APP启动后的第一时间做检查并下载新版插件，比如当前网速是每秒20k而插件的大小是600K,意味着需要等待30s用户才能做下一步操作，很明显这个时间是非常不合理的。有什么办法做一个超时触发？当然有！

于是，我在开始下载的位置获取了系统的时间戳，这个时间戳是从1970年开始算的毫秒时间戳，并且众所周知这个时间戳可以转换成对应的日期小时分钟秒毫秒

long startTime = System.currentTimeMillis;

在每次读取服务器文件流时，做了一个减法计算耗时，

long delyTime = System.currentTimeMillis - startTime

判断delyTime是否大于20s，如果大于20s，触发一个下载超时，取消本次更新。

OK，从上文看，没有任何问题，非常完美解决了问题。

大神说了，看优秀框架的源码才能真正进步。前些天不小心看了Picasso的源码，看到into(Callback callback)方法里，RequestCreator的fetch()方法有这么一行，奇怪这玩意是啥，

long started = System.nanoTime();

然后Request有这么一个方法， 差不多懂了

private static final long TOO_LONG_LOG = TimeUnit.SECONDS.toNanos(5);

String logId() {
    long delta = System.nanoTime() - started;
    if (delta > TOO_LONG_LOG) {
          return plainId() + '+' + TimeUnit.NANOSECONDS.toSeconds(delta) + 's';
    }
    return plainId() + '+' + TimeUnit.NANOSECONDS.toMillis(delta) + "ms";

  }

进一步阅读java API发现：

public static long nanoTime()

返回最准确的可用系统计时器的当前值，以毫微秒为单位。

此方法只能用于测量已过的时间，与系统或钟表时间的其他任何时间概念无关。返回值表示从某一固定但任意的时间算起的毫微秒数（或许从以后算起，所以该值可能为负）。此方法提供毫微秒的精度，但不是必要的毫微秒的准确度。它对于值的更改频率没有作出保证。在取值范围大于约 292 年（263 毫微秒）的连续调用的不同点在于：由于数字溢出，将无法准确计算已过的时间。

例如，测试某些代码执行的时间长度：

long startTime = System.nanoTime();
// ... the code being measured ...
long estimatedTime = System.nanoTime() - startTime;

返回：
系统计时器的当前值，以毫微秒为单位。注意这里说的豪微秒，也就是纳秒，

到这里发现前面觉得完美的代码有什么问题了吗，没错将日期对应的毫秒时间戳改成精确计时的纳秒时间能更准确计算出代码执行的耗时。
